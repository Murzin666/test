# Интеграция через Универсальную платёжную систему Tilda

Официальный протокол Tilda для подключения платёжных систем, которых нет в
готовом списке. Сервер поддерживает сразу несколько торговых точек — у
каждой свой `shop_id` и свой набор URL, см. `multi-tenant.md`.

## Как это работает

1. Покупатель нажимает «Оформить заказ» в корзине ST100.
2. Браузер покупателя автоматически отправляется (POST) на ваш **API URL** —
   эндпоинт `/tilda/checkout` на вашем сервере — с данными заказа и подписью.
3. Сервер проверяет подпись, создаёт заказ в банке и переадресует браузер на
   страницу оплаты банка.
4. После оплаты банк возвращает покупателя на `/tilda/return` — сервер
   проверяет статус в банке и, если оплата прошла, сам отправляет запрос на
   **URL для уведомлений**, который дала Tilda — только после этого заказ в
   CRM Tilda помечается оплаченным.

## Настройка в личном кабинете Tilda

Настройки сайта → Платёжные системы → Универсальная платёжная система →
шаблон «Новая платёжная система (для разработчиков)». Точные значения полей
были согласованы в переписке с Claude при настройке — коротко:

- **API URL**: `https://<ваш-домен-на-railway>/tilda/<shop_id>/checkout`
  (`shop_id` — идентификатор именно этой точки, см. `multi-tenant.md`;
  команда `manage_tenants.py add` печатает готовую ссылку)
- **Поля**: `login`, `currency`, `language`, `country`, `notify_url`,
  `order_id`, `order_desc`, `order_amount`, `order_time`, `client_email`,
  `client_phone`, `client_name`, `signature`, `products` (с подполями
  `index`/`name`/`quantity`/`price`/`total_price`)
- **Подпись заказа**: особые правила `{{login}}{{order_id}}{{order_amount}}`,
  SHA-256, секрет как HMAC-ключ, те же правила для уведомления
- **Уведомление**: признак успеха — поле `paid`, значение `1`; идентификатор
  транзакции — поле `transaction_id`

После сохранения Tilda покажет реальный **URL для уведомлений** — его нужно
передать в `--notify-url` при добавлении этой точки командой
`manage_tenants.py add` (см. `multi-tenant.md`).

## Настройки торговой точки (не переменные окружения!)

Если раньше вы использовали одну точку через переменные `TILDA_LOGIN`,
`TILDA_ORDER_SECRET`, `TILDA_NOTIFY_URL`, `BANK_LOGIN` и т.п. — теперь эти
значения хранятся в БД для каждой точки отдельно, добавляются командой
`manage_tenants.py add shop1 ...`. Подробности и Railway-специфика (Volume,
ключ шифрования) — в `multi-tenant.md`.

## Как протестировать

1. Задеплойте сервер, добавьте точку через `manage_tenants.py add shop1 ...`
   (см. `multi-tenant.md`), вставьте выданный API URL в форму на Tilda.
2. Опубликуйте страницу с корзиной на Tilda.
3. Оформите тестовый заказ. Ожидаемо: браузер уходит на
   `/tilda/shop1/checkout` (мгновенный редирект, можно не заметить) и сразу
   же на страницу оплаты банка (`devpg.centrinvest.ru:8143/flex?...`).
4. Оплатите тестовой картой банка.
5. После оплаты вернётесь на `/tilda/shop1/return`, а оттуда — на страницу
   `tilda_success_url`, если она указана при добавлении точки.
6. Проверьте в CRM Tilda (Мои сайты → CRM), что заказ помечен оплаченным.
   Если нет — проверьте логи сервера на ошибку подписи или недоступность
   notify-url этой точки.
