# Несколько торговых точек на одном сервере

Один и тот же сервер теперь обслуживает сколько угодно торговых точек.
У каждой — свой `shop_id` (короткий идентификатор, который вы придумываете
сами, например `shop1`, `cafe-vostok`) и свой персональный набор URL:

```
API URL для формы Tilda:      https://<ваш-домен>/tilda/<shop_id>/checkout
Возврат покупателя после оплаты: https://<ваш-домен>/tilda/<shop_id>/return  (настраивается сервером автоматически)
```

Настройки самой точки (логин/пароль банка, terminalId, секрет Tilda) хранятся
в базе данных SQLite, а не в переменных окружения — так что добавление новой
точки не требует переменных Variables и передеплоя, только команду CLI.

## Почему БД, а не файл .env / TENANTS_JSON

- Переменные окружения быстро становятся нечитаемыми при десятке точек.
- Чувствительные поля (пароль банка, секрет Tilda) в БД **зашифрованы**
  (см. `app/crypto.py`, `Fernet`/AES). Ключ шифрования (`ENCRYPTION_KEY`)
  хранится отдельно от файла БД — тот, кто получит только файл БД (например,
  из бэкапа), не расшифрует секреты без ключа.

## Шаг 1 — Railway Volume (обязательно!)

Контейнер на Railway эфемерный — файл SQLite пропадёт при каждом новом
деплое, если не подключить постоянный диск:

1. В проекте на Railway откройте вкладку **Volumes** → **New Volume**.
2. Точка монтирования (Mount Path): `/data`.
3. В переменных окружения укажите:
   ```
   DB_PATH=/data/tenants.db
   ```

Без этого шага БД будет обнуляться при каждом деплое — все точки и заказы
придётся добавлять заново.

## Шаг 2 — ключ шифрования

Сгенерируйте один раз и сохраните в переменных окружения Railway:

```bash
python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

```
ENCRYPTION_KEY=<то, что вывела команда>
```

**Потеряете этот ключ — расшифровать сохранённые пароли и секреты будет
невозможно**, придётся добавлять все точки заново с новым ключом.

## Шаг 3 — глобальные переменные (одни на весь сервис)

```
PUBLIC_BASE_URL=https://<ваш-домен-на-railway>
ADMIN_API_KEY=<длинная случайная строка>
ALLOWED_ORIGINS=<домены ваших сайтов на Tilda через запятую>
DB_PATH=/data/tenants.db
ENCRYPTION_KEY=<ключ из шага 2>
```

Обратите внимание: `BANK_LOGIN`, `BANK_PASSWORD`, `TILDA_ORDER_SECRET` и
подобное сюда больше **не входит** — это теперь настройки конкретной точки,
добавляются отдельно (шаг 4).

## Шаг 4 — добавление торговой точки

Через Railway CLI (выполняется в окружении Railway, с доступом к Volume и
ENCRYPTION_KEY):

```bash
railway run python manage_tenants.py add shop1 \
  --bank-login "+79040001313" \
  --bank-password "секрет_банка" \
  --terminal-id 279 \
  --tilda-secret "секрет_из_формы_tilda_для_этой_точки" \
  --notify-url "https://forms.tildaapi.one/payment/custom/psXXXXXXX" \
  --success-url "https://shop1.tilda.ws/thanks" \
  --fail-url "https://shop1.tilda.ws/payment-failed"
```

(`--terminal-id` используется и как логин для формы Tilda — отдельный
`--tilda-login` больше не нужен, эти значения на практике всегда совпадают.)

Команда выведет готовый **API URL**, который нужно вписать в форму
Универсальной платёжной системы именно этой точки на Tilda.

Другие полезные команды:
```bash
railway run python manage_tenants.py list          # список всех shop_id
railway run python manage_tenants.py show shop1     # настройки точки (секреты замаскированы)
railway run python manage_tenants.py remove shop1   # удалить точку
```

Если у Railway CLI нет доступа или не установлен — тот же скрипт можно
запустить локально, указав `DB_PATH` и `ENCRYPTION_KEY` в своём `.env`, но
тогда БД будет создана локально, а не на Volume сервера — так делать смысл
имеет только для теста, не для реальной точки.

## Каждая точка настраивается в Tilda отдельно

Для каждой новой торговой точки в её собственном аккаунте/сайте Tilda нужно
заново пройти настройку из `universal-payment-system.md`, но с уникальным
API URL (`/tilda/shop2/checkout` вместо `/tilda/shop1/checkout`) и **своим**
секретом подписи, который вы передадите в `--tilda-secret` при добавлении
точки.

## Проверка

```bash
curl https://<ваш-домен>/api/health
# {"ok":true,"tenants":["shop1","shop2"]}
```

Список `tenants` в ответе — все точки, которые сервер видит в БД прямо
сейчас.
