# Налоговые атрибуты чека (ФФД) по товарам

Сервер сам собирает блок `receipt` для банка на основе данных заказа.
Источник атрибутов `taxRate`/`type`/`mode` по каждому товару —
**автоматическое поле `Receipt`**, которое Tilda сама добавляет в запрос,
если у товара заполнена вкладка «НДС, ФФД» в карточке товара. Сервер
переводит её словарь (`tax`/`payment_object`/`payment_method`) в коды банка
через таблицы в `app/ffd_mapping.py`.

**Если для товара нет данных (вкладка «НДС, ФФД» не заполнена) или значение
не удалось сопоставить с кодом банка — соответствующее поле просто не
включается в позицию чека.** Сервер сознательно не подставляет своё
значение вместо того, что имел в виду продавец — банк в этом случае
применит свой собственный дефолт, зарегистрированный на терминале, а не
наше предположение.

## Как это работает

1. Tilda присылает состав корзины в поле `products` (JSON-массив с
   `name`/`quantity`/`price`) и, если заполнена вкладка «НДС, ФФД» хотя бы у
   одного товара, отдельное поле `Receipt` — JSON вида:
   ```json
   {"items": [{"name": "Груши", "tax": "vat5", "payment_method": "full_prepayment", "payment_object": "commodity"}], "taxation": "osn"}
   ```
2. Сервер сопоставляет позиции `products` и `Receipt` **по порядку** (не по
   названию!) — так надёжнее для товаров с вариантами (цвет/размер), у
   которых название в `Receipt` может отличаться (`"Apples, Color=Green apples"`
   вместо `"Apples"`).
3. Для каждого атрибута (`taxRate`/`type`/`mode`) отдельно: если он
   определился из данных Tilda — отправляем банку именно его; если нет —
   просто не включаем это поле в позицию, банк подставит свой дефолт.
   Сервер пишет предупреждение в лог для каждого неопределившегося
   атрибута — стоит проверять логи после запуска нового товара в продажу.
4. Если у точки не указан ИНН — чек вообще не отправляется банку (банк
   требует ИНН в каждом запросе с чеком), сервер продолжит работать, просто
   без блока `receipt` — банк применит свои настройки терминала по умолчанию.

## Коды `taxRate` (ставка НДС)

Формат зависит от поля **«ОФД»** в настройках точки (`--ofd-provider`,
по умолчанию `orange_data`) — у разных точек может быть подключён разный ОФД.

**Orange Data** (`orange_data`) — целые числа. После реформы НДС коды
сдвинулись (раньше «1» означало 18%, теперь «1» означает 22%):

| Код | Значение |
|---|---|
| 1 | 22% |
| 2 | 10% |
| 3 | расч. 20/120 |
| 4 | расч. 10/110 |
| 5 | 0% |
| 6 | не облагается |
| 7 | 5% |
| 8 | 7% |
| 9 | расч. 5/105 |
| 10 | расч. 7/107 |

**БИФИТ Онлайн** (`bifit`) — строковые теги, напрямую называющие ставку
(`WITHOUT_VAT`, `VAT_0`, `VAT_5`, `VAT_7`, `VAT_10`, `VAT_20`, `VAT_22`,
`VAT_105`, `VAT_107`, `VAT_110`, `VAT_118`, `VAT_120`, `VAT_122`) — эта
таблица надёжнее Orange Data, у неё нет проблемы со сдвигом кодов после
налоговой реформы, так как значение прямо в названии тега.

Обе таблицы — в `app/ffd_mapping.py:TAX_TO_TAX_RATE_BY_OFD`. Только 6 тегов
БИФИТ (`WITHOUT_VAT`, `VAT_0`, `VAT_10`, `VAT_20`, `VAT_110`, `VAT_120`)
подтверждены исходной документацией банка напрямую — остальные достроены по
той же схеме именования, стоит сверить с банком перед боевым использованием
новых ставок (5%/7%/22%) на точках с БИФИТ.

`type`/`mode` (признак предмета/способа расчёта) — стандартные федеральные
коды по 54-ФЗ, от ОФД не зависят, единые для обоих провайдеров.

Соответствие строковых значений `tax` от Tilda (`vat5`, `vat22` и т.п.) этим
кодам — в `app/ffd_mapping.py:TAX_TO_TAX_RATE`. Таблица получена не из
официальной документации напрямую, а подтверждена эмпирически на реальных
чеках (см. переписку) — для значений `vat18`/`vat20`/`vat118`/`vat120`/`vat122`
однозначного кода нет, поле `taxRate` в этом случае просто не отправляется.

## Коды `type` (признак предмета расчёта)

| Код | Значение |
|---|---|
| 1 | Товар |
| 2 | Подакцизный товар |
| 3 | Работа |
| 4 | Услуга |
| 10 | Платёж |
| 11 | Агентское вознаграждение (если продаёте как агент/комиссионер) |
| 13 | Иной предмет расчёта |

Соответствие значений `payment_object` от Tilda (`commodity`, `service` и
т.п.) — в `app/ffd_mapping.py:PAYMENT_OBJECT_TO_TYPE`.

## Коды `mode` (признак способа расчёта)

| Код | Значение |
|---|---|
| 1 | Предоплата 100% |
| 2 | Частичная предоплата |
| 3 | Аванс |
| 4 | Полный расчёт |
| 5 | Частичный расчёт и кредит |
| 6 | Передача в кредит |
| 7 | Оплата кредита |

Соответствие значений `payment_method` от Tilda (`full_prepayment`,
`full_payment` и т.п.) — в `app/ffd_mapping.py:PAYMENT_METHOD_TO_MODE`.

## Признак агента

Отдельного поля для этого у банка нет — если товар продаётся по агентской
схеме, для него нужно значение `type = 11` («Агентское вознаграждение»),
которое должно прийти от Tilda через `payment_object`. Если такого
сопоставления нет — поле `type` для этой позиции не отправляется, банк
применит дефолт терминала.

## Система налогообложения (СНО)

`taxSystemCode` отдельным полем при добавлении точки не задаётся — сервер
сам вычисляет её из поля `taxation`, которое присылает Tilda (см. таблицу
`TAXATION_TO_TAX_SYSTEM_CODE` в `app/ffd_mapping.py`), и передаёт банку
только если сопоставление удалось. Если Tilda не прислала `taxation` или он
не распознан — поле просто не включается в запрос, банк подставляет СНО,
зарегистрированную на терминале.

## Обязательные поля точки

- `--inn` — ИНН, без него банк не примет чек (не имеет дефолта).
